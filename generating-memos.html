The Slam Dunk: Automating Memos

Hold on a minute. Youre telling me that theres yet another tool that automates memos? What happened to AutoMemo? And what happened to the AutoMemo that came up a couple years before that AutoMemo?

Lets make one thing clear: this is not another AutoMemo. Having developed the recent version, the source code quickly became out of hand because it was trying to do too much on the users behalf. The end users convenience slowly became less attainable at the expense of the source codes complexities. After every problem solved, there was another bigger, more important problem that the poor code I wrote could not keep up with.

What good is this project if I authored the last one? The scope of the source code is drastically different:

  1. It is much smaller. The most recent version of AutoMemo consisted of 2,067 lines of code. (I knew it was bad but I didnt know it was that bad.) Compare that to the version in rdww.extensions which only requires 103 lines (95%95% less).

  2. More importantly, the scope is much smaller. There is very little this version tries to accomplish outside of the bare essentials (putting text in a word document). Users can of course configure it to do more than just that, but it is no longer a priority for the source code to make that decision for you.

How is this possible?

Connecting the Dots

Think about connecting dots as relaying information. The fewer the dots, the more easily you can remember where you were when you started. As the dots increase and there are more twists and turns, its harder to remember how you got where you are (let alone figure out where you need to go).

The closer the beginning and ending of a process are, the better. By limiting the processes that occur in-between, youre limiting the number of errors that can occur. And should any errors arise, well now you know it can only be one of a few.

How does this relate? The previous version(s) of AutoMemo tried to bridge a gap between two very far away points (a gap containing 2067 dots, to be exact). Even while using the app myself, I had to trace back warnings or format things in specific ways to prevent errors in the upcoming pages.

An automated service should be as close to the source of truth as possible. The Data Request Form. This document alone contains the majority of the information required in a memo anyway. Now if we pair this with a package developed to work with rdww.extensions, were only left with applying the finishing touches.

I think integrating memos into rdww.extensions is the greatest example of our goal. The goal of this package(s) is not to be technically-savvy. The goal is to be humanly-intuitive. We could have developed this package to make all these decisions on your behalf, to automate a process that is almost always slightly incorrect. Or, we could develop a package (packages, to be exact) that are adaptive and can suit your every needs, only offering solutions where necessary. Instead of giving you a daunting number of dots to connect on your own with the occassional number missing, were giving you a straight line and clearly labeling them just to make sure you know where to go.

Showcase: Generating Your First Memo

One other reason why there are much fewer lines in the source code is that there is no Shiny application (yet) masking the low-level functions. The intentions of the Shiny application were plentiful, the most important of which was to remove the need for the user to code a memo themselves.

The examples below show quite the opposite. No Shiny application. Pure code.

We hope that this code is intuitive and simple to call/remember. Dont worry, there will be plenty of supporting documents for you to reference in the future. For now, try to soak in the big picture.

### What is this function?
# rdww_memo(memo, output_dir = ".", ...)

### How do I access this function?
# load the library          ->  library(rdww.extensions)
# load the rdwwverse        ->  library(rdwwverse)
# or, reference it directly ->  (as seen below)

rdww.extensions::rdww_memo()
#> ./CLIENT_NAME - YEAR_END_DATE - 1200A Scoping Memo.docx

rdww.extensions::rdww_memo(memo = "Summary Memo")
#> ./CLIENT_NAME - YEAR_END_DATE - 1200B Summary Memo.docx

rdww.extensions::rdww_memo(
  memo = "Procedures Memo",
  output_dir = "path/to/documentation"
)
#> path/to/documentation/CLIENT_NAME - YEAR_END_DATE - Procedures Memo.docx

rdww.extensions::rdww_memo(
  memo = "Controls Over Audit Tools Memo",
  output_dir = "random/path",
  client_name = "Name of Client",
  year_end_date = "11-07-1999"
)
#> random/path/Name of Client - 11-07-1999 - 22852 Memo.docx

rdww.extensions::rdww_memo(
  memo = c("Scoping Memo", "Procedures Memo", "Summary Memo"),
  output_dir = "random/path",
  client_name = "Name of Client",
  year_end_date = "11-07-1999"
)
#> random/path/Name of Client - 11-07-1999 - 1200A Scoping Memo.docx
#> random/path/Name of Client - 11-07-1999 - Procedures Memo.docx
#> random/path/Name of Client - 11-07-1999 - 1200B Summary Memo.docx

Whats so special about returning the path(s) to the memo(s) I want to generate? Lets review what just happened:

  • We generated the path to a scoping memo (the default) with one function call

  • We generated the path to a summary memo with one parameter

  • We specified the path to a procedures memo with another parameter

  • We integrated the client name and year end date with two intuitive parameters

  • We replicated all of that for multiple memos

This is the power of rdww.extensions::rdww_memo(). Think of where that client name is coming from. Think of where the year end date is coming from. Think of all the other information we can extract from the Data Request Form, the analytic itself, or through minimal parameters like the ones above. All that needs to happen is flipping the switch to output Word documents using the paths you created above using the same exact code you wrote above.

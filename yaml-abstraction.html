The Wrong Debate

Admittedly, Ive fallen victim for the benchmark fallacy - validating a software product by its runtime and only its runtime. I mean, if something can run much faster than its counterparts, why wouldnt I ditch everything I know to adopt this product immediately! Runtime metrics have their merit, but it shouldnt be all a product has to offer. More than time saved in compute resources (something we really cant control), why not prioritize time saved in human resources (something we can control)?

Think of it this way. A script will run the same amount of time no matter who clicks Run. Developing a script will take a drastically different amount of time depending on whos typing it.

How long a program takes to run is important; it becomes very important as the number of processes you run increases. But if the only outcome of a faster runtime is it takes less time to learn about a mistake I could have prevented before running the script, did I save any time to begin with?

The Proposed Solution

Consider the following example. You receive data to run an analytic. You start by opening the data and scanning it to quickly check if it looks right. Maybe you modify some column names, trim some white spaces, and consider the data cleaning process over. You run the analytic and wait for the program to tell you what you miss. Rinse and repeat until the analytic is complete.

I cant say I disagree with this approach. Rather than trying to get everything correct on your own, do as much as you can and let the computer tell you what is incorrect. My question is how does the computer know what is correct or incorrect? The computer only knows what we tell it. If we tell it nothing, then it knows nothing. In order for it to know everything then we need to tell it everything good luck finding someone who wants to do that.

An approach I can agree with is this: spend some time telling the computer what you require at the start of a process. Then, the computer can tell you what you need to fix, such that the first time you clean the data is (theoretically) your last. No need to remember any data quirks or ask the computer why wont this error just go away? If we define a set of rules, then the computer - whose sole purpose is to follow rules - will tell us what is and is not our concern.

By separating what I need to focus on from what I do not need to focus on, the task at hand has drastically become much more manageable than had I tried to fix everything on my own. Chances are it takes less time as well.

Abstraction & Separation of Concerns

We first have to tell the computer what it needs to know. There are many ways of approaching this problem, and rdww.yaml settled for - big surprise - YAML files. The benefits are two-fold:

  • YAML files are very easy for computers to read

  • YAML files are very easy for humans to read (and continuously update/maintain)

A third benefit that Ill discuss here is that it makes the code in the rest of the rdwwverse very easy to write. All objects (analytics, datasets, memos, etc.) now follow a defined structure, a structure that is expected everywhere else in the ecosystem. This leads to deterministic behavior when interacting with objects, reducing a users concerns about how a process handles any object.
